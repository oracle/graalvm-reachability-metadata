# Post-generation intervention report

Library: org.springframework:spring-core:5.3.15
Stage: metadata_fix_failed

## Summary

The Codex metadata-fix run reduced the native test failures but did not fully resolve the generated support for `org.springframework:spring-core:5.3.15`. The initial native run reported 19 failures after Codex's metadata edits. I removed generated tests whose failures were not reachability-metadata problems and preserved the tests that still expose missing metadata.

After the intervention, `./gradlew test -Pcoordinates=org.springframework:spring-core:5.3.15` still fails with 5 metadata-related native runtime failures:

- `ConfigurableObjectInputStreamTest.resolvesProxyClassWithObjectInputStreamDefaultClassLoader()` fails with `ClassNotFoundException: org.springframework.core.Ordered` from `ObjectInputStream.resolveProxyClass`. The generated metadata does not include the reflective class lookup and proxy support needed for the `org.springframework.core.Ordered` / `java.io.Closeable` proxy-interface resolution path.
- `SerializableTypeWrapperInnerFieldTypeProviderTest.restoresFieldProviderDuringDeserialization()` fails with `MissingReflectionRegistrationError` for `public java.lang.Object()` during `ObjectInputStream` deserialization.
- `SerializableTypeWrapperInnerMethodParameterTypeProviderTest.restoresMethodParameterProviderForConstructorDuringDeserialization()` fails with the same missing `java.lang.Object` constructor registration during `ObjectInputStream` deserialization.
- `SerializableTypeWrapperInnerMethodParameterTypeProviderTest.restoresMethodParameterProviderForMethodDuringDeserialization()` fails with the same missing `java.lang.Object` constructor registration during `ObjectInputStream` deserialization.
- `SunReflectionFactoryHelperTest.createsInstanceWithoutRunningTargetConstructor()` fails with the same missing `java.lang.Object` constructor registration through `SunReflectionFactoryInstantiator`.

Codex attempted to address the `java.lang.Object` constructor miss by adding exact and broader constructor reflection registrations, but the native run continued to report the same missing registration. It also fixed earlier `ResolvableType` field misses, so the remaining failures are a smaller set of unresolved metadata issues rather than broad test-suite breakage.

## Non-metadata failures removed

I removed or trimmed generated tests whose root cause was not missing reachability metadata:

- Removed CGLIB runtime code-generation tests based on `Enhancer`, `Proxy`, `MethodProxy`, `Mixin`, `AbstractClassGenerator`, `ConstructorDelegate`, `MethodDelegate`, and `DuplicatesPredicate`. These failed because the generated tests exercised CGLIB bytecode generation / `KeyFactory` behavior that is not a reachability-metadata miss in this native-image environment.
- Removed the generated `BridgeMethodResolverTest` method that used `Enhancer`, while preserving the pure bridge-method resolver coverage.
- Removed the generated `ReflectUtilsTest` assertion around `ReflectUtils.findInterfaceMethod(Ordered.class)`, which failed as a behavioral/test-assumption mismatch rather than missing metadata.
- Removed `DefaultSerializerTest` and `DefaultDeserializerTest` methods that serialized/deserialized a generated `LinkedHashMap` payload. That payload requires application/JDK serialization configuration and is not spring-core library metadata. The remaining serializer/deserializer tests still cover Spring's serializer behavior with supported payloads.
- Removed the generated `ClassUtilsTest` method that asserted a concrete declaring class for `getMostSpecificMethod`; the native behavior returned the interface method, so this was a generated assertion mismatch, not missing metadata.

## Why the remaining generated support should be preserved

The remaining 114 passing native tests cover a broad set of spring-core behavior: class/resource loading, annotation synthesis, conversion utilities, ASM usage, reflection helpers, object instantiation strategies, serialization helpers, and VFS utilities. These tests are not scaffold-only; they exercise reachable spring-core APIs and validate that the metadata already generated by Codex enables substantial native-image coverage.

The 5 remaining failures identify specific unresolved metadata gaps: proxy/class lookup metadata for `org.springframework.core.Ordered` and stubborn reflection metadata for `java.lang.Object.<init>()` on Spring serialization and Objenesis paths. Because these are metadata-related, the corresponding generated tests should be kept to document and reproduce the missing support instead of being removed.
